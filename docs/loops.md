# Loops

Loops are the operating spine of Executive Council, the product developed by Anyang Intelligence.

Executive Council is not just memory, a dashboard, or a chat interface. It is a set of living loops that turn context into decisions, decisions into coordinated action, action into evidence, and evidence into better future judgment.

## Canonical Executive Loop

Every installation should inherit this loop:

```text
memory
  -> decision
  -> coordination
  -> review
  -> learning
  -> updated memory
```

This is the core recursive structure of Executive Council.

Executive Council is the loop, not merely the agent, repository, or interface. Those are components of the governed operating cycle. The cycle turns human intent and AI synthesis into authorized action, evidence, review, and improved future judgment.

- `memory` preserves facts, assumptions, commitments, risks, and open loops.
- `decision` turns ambiguity into options, tradeoffs, recommendations, and human approval points.
- `coordination` turns approved decisions into owners, next actions, dependencies, deadlines, and status.
- `review` compares the current state against priorities, risks, and commitments.
- `learning` compares expected outcomes with actual outcomes and updates the operating model.
- `updated memory` prevents the organization from relearning the same lesson from scratch.

## Loop Grammar

Every loop should define:

- **Signal:** What starts the loop.
- **Memory objects:** What the loop needs to remember.
- **Decision:** What judgment the loop prepares.
- **Action:** What changes in the world.
- **Evidence:** What proves the action happened or what result occurred.
- **Review cadence:** When the loop is revisited.
- **Learning update:** What changes for the next cycle.
- **Governance boundary:** What requires human, owner, parent, board, CPA, attorney, or other professional approval.

If a loop cannot name these elements, it is not yet operational.

## Loop Types

### Operating Loops

Operating loops run customer work.

Examples:

- Book Club: book selection, meeting, discussion, memory, next selection.
- Learning Core: learner signal, parent review, learning plan, experience, evidence, reflection, adjustment.
- Grace Gems: listing signal, listing gate, publish/revise/hold, customer response, marketplace update.
- Media Production: idea, brief, production, quality gate, publish, performance review, learning.
- Mountain Villa: property signal, risk rank, mitigation, proof, seasonal readiness update.

### Governance Loops

Governance loops prevent Executive Council from acting beyond authority.

Examples:

- Owner approval before Grace Gems product, pricing, policy, shipping, refund, certificate, or authenticity claims.
- Parent/guardian review before Learning Core child-facing learning decisions.
- CPA/bookkeeper review before final tax, withholding, payroll, nonprofit, contractor, or asset classification.
- Human approval before Media Production publication, client commitments, rights decisions, spending, or hiring.

Governance loops should slow the system down when stakes rise.

### Learning Loops

Learning loops improve domain judgment.

They ask:

- What did we believe?
- What happened?
- What surprised us?
- What should change next time?
- What should be preserved in memory?

The goal is better judgment without blame theater.

### Recursive Improvement Loops

Recursive improvement loops improve Anyang Intelligence itself.

Examples:

- `coffee` extracts one learning from the last work cycle and proposes one system improvement.
- Customer comparison identifies shared primitives across lanes.
- Portfolio knowledge synergies convert customer lessons into reusable patterns.
- Skills encode recurring procedures so future agents do not rely on memory alone.

The recursive improvement loop is:

```text
work cycle
  -> friction or learning noticed
  -> repo state updated
  -> skill / template / guardrail / doc improved
  -> next work cycle becomes smarter
```

## Cadence

Cadence is the heartbeat of loops.

Different domains need different rhythms:

- Daily: learner plans, urgent operations, high-change customer signals.
- Weekly: executive reviews, marketplace review, production review, property risk review.
- Monthly: book club, nonprofit leadership review, portfolio review, donor/material stewardship.
- Seasonal: Mountain Villa readiness, school-year planning, campaign cycles, grant/reporting cycles.
- Event-driven: payment received, contractor hired, customer complaint, safety risk, owner approval, board decision, tax-sensitive question.

The right cadence is the one that catches meaningful change before it becomes drift.

## Evidence

Loops become trustworthy when they produce evidence.

Common evidence:

- Decision memo.
- Operating review.
- Risk register update.
- Payment record.
- Contract or invoice.
- Owner approval.
- Parent/guardian approval.
- Board approval.
- Listing/order export.
- Proof photo.
- Portfolio artifact.
- Published asset.
- Customer transcript.
- Performance metric.
- Lesson learned.

Without evidence, a loop becomes a story the system tells itself.

## Failure Modes

Watch for these loop failures:

- **Open-loop drift:** commitments exist but no review date or owner exists.
- **Memory decay:** lessons are learned in conversation but not preserved.
- **Cadence mismatch:** reviews happen too often to be useful or too rarely to catch change.
- **Governance bypass:** the system treats a recommendation as authority.
- **Evidence gap:** conclusions are recorded without receipts, source notes, approvals, or artifacts.
- **Overbuilt loop:** a lightweight context becomes bureaucratic.
- **Underbuilt loop:** a high-trust or high-risk context lacks review, approval, or escalation.
- **No recursive update:** the system notices friction but does not improve its own docs, skills, templates, or guardrails.

## Product Implication

Loops should become a first-class product primitive.

Every project should eventually be expressible as:

```text
loop name
signal
memory objects
decision prepared
action path
evidence required
cadence
review questions
learning update
governance boundary
```

Project-specific loops describe the domain. Shared loop infrastructure becomes the software spine.

Loops move work forward. [Membranes](membranes.md) decide what may move across customer or domain boundaries.
