# Executive OS Install Method

## Purpose

The install method is the repeatable pattern behind every Anyang Intelligence customer installation.

Each customer has a different domain, but the Executive OS should always answer the same core question:

What does this operating context need to remember, decide, coordinate, review, and learn?

## Seven-Part Install Pattern

### 1. Context Map

Define the operating context before designing workflows.

Capture:

- What the organization or asset is.
- Who it serves.
- What work it performs.
- What cadence governs it.
- What constraints shape decisions.
- What risks or failure modes matter most.

The context map prevents the Executive OS from becoming generic.

### 2. Executive Memory Objects

Define what the system must remember.

Memory objects are the durable nouns of the operating context: customers, students, listings, programs, donors, risks, books, productions, properties, projects, owners, decisions, assumptions, and follow-ups.

The goal is not to store everything. The goal is to preserve the information leadership repeatedly needs to reconstruct.

### 3. Decision System

Define the decisions the Executive OS should prepare.

Every installation should identify:

- Recurring decisions.
- High-consequence decisions.
- Decisions that currently depend on scattered context.
- Decisions that need evidence, options, tradeoffs, or approval.

The Executive OS prepares decisions. Humans retain authority.

### 4. Operating Review Cadence

Define how the operating context gets reviewed.

The review may be weekly, monthly, seasonal, campaign-based, or event-driven. The point is to keep the system alive.

Every review should answer:

- What changed?
- What is open?
- What is blocked?
- What risks are rising?
- What decisions are needed?
- What should happen next?

### 5. Risk and Governance Boundary

Define what the Executive OS may and may not do.

Every installation needs explicit boundaries for:

- Human approval.
- Sensitive information.
- External communication.
- Financial commitments.
- Legal, medical, educational, or safety-sensitive decisions.
- Evidence standards.
- Escalation paths.

The higher the trust level, the narrower the AI authority should be.

### 6. 30-Day Installation Plan

Define the first month of implementation.

The default month should move from legibility to action:

- Week 1: Context and source map.
- Week 2: Memory and decision workflow.
- Week 3: Operating review and risk boundary.
- Week 4: First review, first learning loop, and next-cycle plan.

Customer-specific installs can expand this into a four to six week sprint when needed.

### 7. Success Criteria

Define how the installation proves value.

Success criteria should be observable, not decorative.

Good criteria answer:

- Is context easier to reconstruct?
- Are decisions clearer?
- Are follow-ups less likely to disappear?
- Are risks visible earlier?
- Are reviews useful enough to repeat?
- Is the organization learning from its own outcomes?

## Output Standard

Every customer installation should include:

- A `README.md` describing the domain, Executive OS role, operating thesis, and install link.
- An `executive-os-install.md` describing the installation method for that domain.

Use [templates/customer-install/](../templates/customer-install/README.md) as the starting point for new customer folders.

## Product Implication

The install method is the bridge from service to software.

Repeated patterns across customer installs become product primitives. Customer-specific details stay as configuration, templates, or vertical modules.
