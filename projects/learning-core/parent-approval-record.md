# Parent Approval Record

This is the internal/operator record for what the parent or guardian approved, what remains provisional, and what changed after review.

Use it after a **30-Day Personalized Learning Plan** draft exists and before the plan is treated as ready for use.

This record makes the authority layer visible. A draft may be clear and useful, but it is not the same thing as parent approval.

## Purpose

Use this record to:

- preserve what the parent approved
- preserve what the parent declined or changed
- keep provisional items visible
- record what may be saved
- record what may be shared
- prevent a draft from being mistaken for a final decision

## Operating Loop

This artifact sits inside the Learning Core loop:

```text
observe
  -> gather insight
  -> recommend next move
  -> parent reviews
  -> system updates
```

The approval record is the clearest evidence that the `parent reviews` step actually happened.

## When To Use

Use this record when:

- a `Ready` draft is returned for approval
- a `Provisional` draft is returned for approval with assumptions
- the parent requests changes before use
- the operator needs to preserve what was approved versus what is still pending

Do not treat the existence of a draft, exemplar, evidence map, or checklist as approval by itself.

## Minimal Approval Rule

At minimum, the record should show:

- who approved the draft for use
- what the parent approved
- what changed from the original draft
- what is still not approved
- whether the plan is safe to use now

If those items are unclear, the plan should not be treated as fully approved.

## Common Outcomes

### 1. Approved As Sent

Use when the parent approves the draft without changes.

Record:

- all approved items
- safe-to-use status = `yes`

### 2. Approved With Changes

Use when the parent approves the direction but requests edits.

Record:

- approved items
- changed items
- whether the current draft can still be used now or must be updated first

### 3. Still Provisional

Use when the parent accepts the general direction but some items remain open.

Record:

- approved items
- not-yet-approved items
- visible assumptions
- safe-to-use status = `only after listed updates` or `no`

### 4. Hold After Review

Use when parent review reveals a boundary, concern, or disagreement that pauses use.

Record:

- what caused the pause
- what additional input or change is required
- safe-to-use status = `no`

## Update Rule

If the parent changes:

- rhythm
- screen-time budget
- app rule
- reading basket direction
- evidence rules
- watch-item framing
- save/share boundaries

then update the plan and the approval record together.

Do not leave the plan saying one thing and the approval record another.

## Evidence Of Review

Good review evidence may include:

- parent reply notes
- a dated approval summary
- a written change list
- a parent-facing checklist returned with answers

The record does not need to be bureaucratic. It only needs to be clear enough that a future operator can tell what was actually approved.

## Boundary

This record is not:

- legal waiver language
- diagnosis
- proof of mastery
- a substitute for parent judgment
- permission for unsupervised child-facing AI

It is an authority and change-tracking artifact for the Student Operating System.
