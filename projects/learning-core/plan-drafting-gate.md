# Plan Drafting Gate

This is an internal/operator checklist for turning a `Ready` or `Provisional` intake into a **30-Day Personalized Learning Plan** draft without inventing learner facts.

Use this only after the intake has already been classified with [onboarding-readiness-checklist.md](onboarding-readiness-checklist.md).

If the case is still `Hold`, stop and use [hold-response-template.md](hold-response-template.md) instead of drafting.

## Core Rule

Every meaningful part of the draft should be traceable to one of these:

- a parent-approved input
- a clearly labeled provisional assumption
- a neutral template default that does not claim learner-specific truth

If a section cannot be supported by one of those three sources, leave it open, mark it as missing, or pause the draft.

## Allowed Inputs

Build the draft only from:

- parent intake answers
- parent-approved student voice notes
- parent-approved schedule and resource notes
- parent-approved privacy and sharing rules
- parent-approved support context
- clearly stated `Provisional` assumptions allowed by [onboarding-readiness-checklist.md](onboarding-readiness-checklist.md)
- safe template defaults from [30-day-plan-template.md](30-day-plan-template.md)

Do not treat mock simulations, example language, or operator intuition as real learner evidence.

## Drafting Status Check

Before filling the template, confirm:

- intake status is `Ready` or `Provisional`
- approval authority is clear
- privacy and sharing boundaries are clear enough to draft safely
- the parent has approved plan drafting from the current intake
- missing inputs are listed explicitly
- any provisional assumptions are listed explicitly

If any of the above is untrue, pause.

## Section-By-Section Gate

Use this map before filling each section of [30-day-plan-template.md](30-day-plan-template.md).

### 1. Plan Status

Allowed:

- `Ready`, `Provisional`, or `Hold` from the actual intake review
- parent goal stated by the parent
- missing-input list drawn from the intake review

Do not:

- upgrade a `Provisional` case to `Ready`
- hide missing inputs

### 2. Parent Authority And Safety

Allowed:

- parent approval boundaries actually named by the parent
- template safety language that preserves parent authority and no-diagnosis/no-mastery boundaries

Do not:

- infer consent to save or share information
- imply outside-support approval if it was not given

### 3. New-Student Onboarding

Allowed:

- onboarding steps already approved or clearly planned
- `complete / planned / skipped for now` statuses grounded in actual intake

Do not:

- mark setup items complete unless the parent actually confirmed them

### 4. Learner Context Summary

Allowed:

- parent-described interests
- parent-described strengths
- parent-described friction or support signals
- parent-described constraints
- evidence quality labeled honestly as `strong`, `medium`, or `thin`

Do not:

- assign reading levels, math levels, diagnoses, or motivational narratives not given by the parent
- convert frustration into a fixed label

### 5. 30-Day Rhythm

Allowed:

- rhythm options supported by the parent schedule
- screen-based activity levels that fit the parent-set screen-time budget, if there is one
- minimum viable, standard, and fuller day structures when the case is `Provisional`
- safe defaults like short sessions, parent observation, and light evidence collection

Do not:

- invent an exact daily schedule if the family did not provide one
- exceed or ignore a parent-set screen-time budget
- overload the month with unsupported specificity

### 6. Weekly Focus Blocks

Allowed:

- week goals tied to the parent goal
- activities grounded in stated interests, available resources, and approved tools
- evidence prompts that match the portfolio rules

Do not:

- claim a week is targeting a learner weakness unless the parent named it and approved that framing
- create activity specificity that depends on unavailable materials or unconfirmed preferences

### 7. Reading Basket Plan

Allowed:

- books or formats actually named by the parent
- template basket structure when the parent approved a general reading basket but did not choose titles yet

Do not:

- pretend exact book matches are known when they are not

### 8. Khan Academy Kids Use Rule

Allowed:

- only the parent-approved use rule
- only the parent-approved screen-time fit
- template default observation language if the app is approved

Do not:

- imply unsupervised use
- treat app time as outside the parent budget unless the parent explicitly said that
- imply app progress is mastery, grading, diagnosis, or curriculum completion

### 9. Portfolio Evidence Plan

Allowed:

- only parent-approved evidence types and save/share rules
- light default suggestions when the parent approved a portfolio but did not specify exact rhythm

Do not:

- choose evidence categories the parent did not permit
- imply Anyang owns the decision about what gets saved

### 10. Watch Items And Support Signals

Allowed:

- only parent-named concerns
- calm pattern language
- safe next-step language that returns authority to the parent

Do not:

- introduce new support concerns
- diagnose
- escalate tone beyond the actual evidence

## Provisional Draft Rules

If the case is `Provisional`:

- label the plan status as `Provisional`
- carry the missing-input list into the draft
- mark assumptions visibly
- prefer lighter structure over false precision
- use open placeholders when exact facts are missing

A provisional draft is a parent review tool, not a disguised final plan.

## Unsupported Section Test

If any section makes you think:

- `I know what they probably mean`
- `This is usually true for kids this age`
- `I can fill this in to make the plan look complete`

stop and remove that line or mark it as missing.

Completeness is less important than truthfulness.

## Pre-Release Check

Before sending the draft for parent review, confirm:

- plan status matches the intake classification
- missing inputs are still visible
- every learner-specific claim came from parent-approved inputs
- every assumption is labeled
- screen-based activities fit the parent-set budget, if one exists
- no pricing, retainer, or internal skill language appears
- no diagnosis, grading, proof-of-mastery, or legal/accountability claim appears
- save/share language matches parent boundaries
- child-facing prompts remain parent-approved

## Handoff Output

When the draft is done, the operator should be able to say:

- what came directly from the parent
- what remains missing
- what assumptions were used
- what parent approvals are still needed before use

If the operator cannot do that, the draft is not ready for review.
