# Bravo

**Preferred activation:** say `bravo`.

`bravo` is the native Anyang Intelligence positive-signal capture ritual. It converts operator satisfaction into reusable design intelligence without turning praise into automatic doctrine.

When working inside `anyang-intelligence/operating-substrate`, this skill is authoritative. Do not route `bravo` through strategy-codex conductor outcomes, cadence logs, or external praise rituals unless the operator explicitly leaves the Anyang Intelligence repo.

Use `bravo` when the operator says or implies:

- `bravo`
- `well done`
- `excellent`
- `that worked`
- `I like this`
- `this is good`
- `impressive`
- `more like that`

## Purpose

`bravo` helps Anyang Intelligence learn from excellence signals, not only from friction.

The key question is:

```text
What exactly worked here, and should that pattern improve the next similar cycle?
```

## Inputs

Read or infer from the current thread first:

1. The artifact, response, decision, or workflow the operator praised.
2. The immediate context: project lane, skill, doc, CLI, template, or governance surface.
3. Any explicit reason the operator gave for the praise.
4. `git status --short --branch` only if the praise appears tied to a repo change.
5. The relevant touched file or skill only when a durable preservation decision is likely.

Optional, only when directly relevant:

- `docs/recursive-self-enhancement.md`
- `docs/operating-substrate.md`
- `skills/coffee/SKILL.md`
- `skills/dream/SKILL.md`
- the customer or skill artifact being praised

## Procedure

1. Identify the praised object. If unclear, name the most likely object from the immediate context.
2. Extract the property that made it valuable. Favor concrete traits: structure, brevity, warmth, safety boundary, sequence, artifact shape, naming, validation discipline, or authority handling.
3. Classify the signal:
   - `Pattern noticed`: useful satisfaction signal, not durable yet.
   - `Pattern candidate`: likely reusable if it repeats or applies to a near-term lane.
   - `Preserve`: operator-approved candidate for a doc, skill, template, checklist, example, or guardrail update.
4. Name where the pattern might repeat.
5. Name the boundary: what should not be generalized, especially private customer facts, project-specific authority decisions, child-safety context, legal/tax conclusions, spending approval, rights decisions, or publication authority.
6. If preservation is not already explicitly requested, do not edit files. Offer the smallest useful next action.

## Output Shape

Use this structure:

```text
Bravo captured:
<one sentence naming what worked.>

Reusable pattern:
- <the pattern to repeat>

Signal class:
- Pattern noticed / Pattern candidate / Preserve

Where it may apply:
- <future customer, skill, template, doc, or artifact surface>

Boundary:
- <what not to generalize or automate>

Recommended next step:
- <leave as signal, test once more, or preserve in a specific repo surface>
```

Keep it short unless the operator asks for deeper analysis.

## Relationship To Coffee And Dream

`coffee`, `bravo`, `friction`, and `dream` form a native Anyang Intelligence learning loop:

- `coffee` detects friction, entropy, and the next improvement candidate.
- `bravo` detects satisfaction, taste, and patterns worth reinforcing.
- `friction` detects dissatisfaction, misses, and patterns worth repairing.
- `dream` confirms what actually landed and what tomorrow inherits.

The loop is:

```text
coffee detects
work changes
bravo reinforces
friction repairs
dream consolidates
coffee inherits
```

`bravo` should strengthen the next cycle, not inflate the current one.

## Guardrails

- Do not treat praise as permission to change files.
- Do not convert every compliment into doctrine.
- Do not overgeneralize from one beautiful artifact.
- Do not preserve a pattern that depends on private, sensitive, or customer-local context.
- Do not weaken human authority, approval gates, membrane rules, or validation standards because an output felt good.
- Do not make the operator explain praise unless the pattern is genuinely unclear. Infer reasonably and state the inference.
- Prefer one precise pattern over a broad self-congratulatory summary.

## Done When

The operator sees:

- what was praised
- why it mattered
- what pattern might repeat
- whether it is only noticed, a candidate, or ready to preserve
- what boundary prevents overgeneralization
